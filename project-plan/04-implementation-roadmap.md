# 技术实施路线与阶段计划

## 1. 第一版技术路线

建议第一版采用静态网页生成方式：

```text
data/families.json
  -> build.py
  -> templates/family.html
  -> docs/f/<slug>/index.html
  -> docs/qrcodes/<id>.png
```

团队只需要维护一份家庭数据文件和对应媒体文件，Python 自动生成网页和二维码。

## 2. 推荐项目结构

```text
suiyue-memory/
├─ data/
│  └─ families.json
├─ templates/
│  └─ family.html
├─ static/
│  └─ style.css
├─ media/
│  ├─ F001/
│  │  ├─ photo.jpg
│  │  ├─ story.mp3
│  │  └─ video.mp4
│  └─ F002/
│     └─ photo.jpg
├─ docs/
│  └─ 自动生成的网站文件
├─ project-plan/
│  └─ 项目方案文档
├─ build.py
├─ requirements.txt

```

其中 `docs/` 建议作为网站发布目录，不手动编辑。

## 3. 分阶段推进

### 阶段一：验证最小闭环

目标：做出 1 个测试家庭页面和 1 个二维码。

交付：

1. `families.json` 中有 1 条测试数据；
2. 家庭页面可显示照片、故事、金句；
3. 生成二维码；
4. 本地浏览器可以打开页面。

验收标准：

> 扫描二维码后能打开对应家庭页面。

### 阶段二：支持声音明信片

目标：让同一个家庭页面支持音频。

交付：

1. 页面出现音频播放器；
2. 无音频时自动隐藏音频模块；
3. 手机端可以播放；
4. 页面不会因为音频字段为空而报错。

验收标准：

> 第一堂课后的二维码不变，第二堂课补充音频后仍打开同一页面。

### 阶段三：支持 AI 影像

目标：让页面支持可选视频模块。

交付：

1. 页面可嵌入 MP4；
2. 无视频时自动隐藏；
3. 页面标明“包含 AI 辅助创作内容”；
4. 视频文件体积控制在适合手机访问的范围内。

验收标准：

> 有授权、有素材的家庭可以显示 AI 记忆影像，其他家庭页面保持简洁。

### 阶段四：批量生成与数据校验

目标：支持 10-20 个家庭稳定生成。

交付：

1. 检查家庭 ID 是否重复；
2. 检查 slug 是否重复；
3. 检查照片、音频、视频路径是否存在；
4. 检查故事长度；
5. 检查展示权限字段是否合法；
6. 错误信息能指出具体家庭编号。

验收标准：

> 一条数据有问题时能及时发现，不会生成错误页面用于现场展示。

### 阶段五：上线与现场使用

目标：部署到 GitHub Pages 或其他静态网站托管。

交付：

1. 正式网址确定；
2. 所有二维码使用正式网址生成；
3. 手机扫码测试通过；
4. 展示授权记录完整；
5. 现场备用访问链接准备好。

验收标准：

> 现场至少 10 个家庭可扫码访问自己的数字档案 V1/V2。

## 4. 暂不做小程序的原因

首轮不建议直接开发微信小程序，因为会增加：

1. 注册与审核；
2. 服务器和数据库；
3. 用户登录与权限；
4. 文件上传和存储；
5. 后续运维；
6. 肖像、声音、未成年人隐私管理压力。

等社区数量增加、作品数量扩大、确实需要家庭登录和权限管理时，再升级到 Flask 后台或小程序更合理。

## 5. 后续升级路径

如果首轮验证顺利，可以按需求升级：

```text
静态网页
  -> Flask + SQLite 管理后台
  -> 家庭访问密码
  -> 文件上传与权限控制
  -> 微信小程序或社区数字记忆平台
```

不要在第一版同时做完所有功能。当前最重要的是把“采集、整理、扫码查看、声音补充”这条链跑通。
